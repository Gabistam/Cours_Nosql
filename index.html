1.1

Contexte & Histoire du NoSQL

Avant de plonger dans les outils, il faut comprendre pourquoi le NoSQL existe. Ce n'est pas un caprice technologique — c'est une réponse directe à des problèmes concrets que les bases relationnelles n'arrivaient plus à résoudre à l'échelle du web moderne.

⚡
Le problème à l'origine de tout

Imaginez que vous gérez un réseau social avec 100 millions d'utilisateurs actifs par jour. Chaque action — un like, un commentaire, une connexion — génère une écriture en base. Les bases relationnelles traditionnelles sont construites pour la cohérence stricte et la complexité des relations. Face à ce volume, elles montrent leurs limites : latence qui explose, montée en charge difficile, schéma rigide qui freine l'évolution produit.

NoSQL n'est pas "mieux" que SQL — c'est une réponse différente à des besoins différents. Le bon développeur sait choisir le bon outil selon le contexte.
📅
Chronologie clé
1998
Le terme "NoSQL" apparaît

Carlo Strozzi l'utilise pour une base open source sans SQL. La définition moderne n'existe pas encore.

2003–05
L'explosion du web fait craquer les bases relationnelles

Google, Amazon, Facebook commencent à atteindre des volumes que MySQL ne peut plus absorber.

2006–07
Bigtable (Google) & Dynamo (Amazon)

Deux articles de recherche fondateurs qui posent les bases architecturales du NoSQL moderne : distribution, tolérance aux pannes, scalabilité horizontale.

2009
"Not Only SQL" — la définition s'impose

Eric Evans popularise la formulation lors d'un meetup à San Francisco. MongoDB est lancé par 10gen. L'écosystème décolle.

2010+
Explosion de l'écosystème

Cassandra (Facebook → Apache), CouchDB, Redis, Neo4j… Chaque modèle trouve son cas d'usage. Le NoSQL devient incontournable dans l'industrie.

🛒 ANECDOTE — Amazon invente (presque) le NoSQL sans le savoir
Dans les années 2000, Amazon réalise que ses bases MySQL ne tiennent pas la charge pendant les soldes. Trop de rigidité, pas assez de scalabilité. Leurs ingénieurs créent un système maison plus souple : Dynamo. Ils n'ont pas inventé le mot, mais ils ont ouvert la voie à ce que l'on appelle aujourd'hui le NoSQL distribué.

1.2

SQL vs NoSQL — Ce qui change vraiment

On oppose souvent SQL et NoSQL comme si l'un était supérieur à l'autre. C'est une erreur de cadrage. Ces deux approches répondent à des philosophies différentes face à des besoins différents. Votre rôle en tant que développeur, c'est de comprendre ces différences pour faire le bon choix.

⚖️
Comparaison directe
Critère SQL (relationnel) NoSQL
Structure des données Schéma fixe et rigide, défini à l'avance Schéma flexible ou absent, évolutif
Modèle de données Tables avec lignes et colonnes typées Clé-valeur, Document, Colonnes, Graphe
Langage de requête SQL — normé, portable entre SGBDR API spécifique à chaque outil, pas de standard
Scalabilité Verticale (scale-up : + de RAM, CPU) Horizontale (scale-out : + de serveurs)
Transactions ACID strict — cohérence forte garantie Cohérence éventuelle (BASE) ou ACID limité
Relations entre données JOINs puissants, clés étrangères Relations limitées ou gérées manuellement
Évolution du schéma Migrations formelles, processus lourd Modification à la volée, très agile
Cas typiques Finance, ERP, e-santé, gestion Web à grande échelle, temps réel, Big Data
🔑
Le concept ACID vs BASE
ACID — SQL
  • Atomicité — tout ou rien
  • Cohérence — état valide garanti
  • Isolation — transactions séparées
  • Durabilité — persistance assurée
BASE — NoSQL
  • Basically Available — toujours disponible
  • Soft state — état peut varier
  • Eventual consistency — cohérence à terme
Concrètement : le nombre de "likes" sur un post peut afficher 1042 sur votre écran et 1043 sur celui de votre voisin pendant quelques millisecondes — c'est de la cohérence éventuelle. Acceptable ici. Pas pour un virement bancaire.
🐦 ANECDOTE — Twitter abandonne MySQL pour survivre
Au début, Twitter utilisait MySQL pour ses timelines. Avec la montée en charge, les requêtes avec JOINs complexes ralentissent tout. Twitter crée Manhattan, un système NoSQL maison capable de gérer des millions de requêtes par seconde sur des centaines de serveurs. Leçon : quand les données sont massives et le besoin de vitesse critique, NoSQL fait la différence.
1.3

Panorama des 4 modèles NoSQL

Il n'existe pas un NoSQL, mais quatre grandes familles, chacune pensée pour un type de problème précis. Comprendre ce panorama, c'est ce qui vous permettra de justifier vos choix d'architecture dans un contexte réel. Dans ce cours, chaque modèle sera associé à un module d'une application e-commerce — SportShop.

🔑
Clé-Valeur
Redis · DynamoDB · Memcached
La structure la plus simple : une clé unique pointe vers une valeur. Extrêmement rapide car tout tient en mémoire. Idéal pour le cache, les sessions et les files d'attente.
Use case SportShop : Panier & Sessions
📄
Document
MongoDB · CouchDB · Firestore
Les données sont stockées sous forme de documents JSON/BSON auto-descriptifs. Structure flexible, données hiérarchiques, idéal pour les catalogues évolutifs.
★ Focus de ce cours · Catalogue produits
🏛️
Colonne
Cassandra · HBase · ScyllaDB
Les données sont organisées par colonnes plutôt que par lignes. Performances d'écriture massives, scalabilité extrême. Conçu pour le Big Data et les séries temporelles.
Use case SportShop : Logs & Analytiques
🌐
Graphe
Neo4j · ArangoDB · Amazon Neptune
Les entités sont des nœuds reliés par des relations typées. Parfait pour modéliser des connexions complexes : réseaux sociaux, systèmes de recommandation.
Use case SportShop : Recommandations
🛒
Le fil rouge : application SportShop

Tout au long de ce cours, vous allez travailler sur une application e-commerce sportive — SportShop. Chaque module fonctionnel de cette application est volontairement associé au modèle NoSQL le plus adapté. C'est ce qu'on appelle la persistance polyglotte : utiliser plusieurs bases de données, chacune dans son domaine d'excellence.

Polyglot Persistence : dans une vraie application moderne, il est courant d'utiliser MongoDB pour le catalogue, Redis pour le cache et les sessions, Cassandra pour les logs, et Neo4j pour les recommandations — le tout en parallèle.
1.4

Le Modèle Document — Fondements

Le modèle document est aujourd'hui le plus répandu dans le développement web moderne. Sa popularité vient d'une idée simple mais puissante : stocker les données dans le même format que celui utilisé dans le code — le JSON.

📄
Qu'est-ce qu'un document ?

Un document est une structure de données auto-descriptive au format JSON (ou BSON pour MongoDB). Contrairement à une ligne SQL qui ne contient que des valeurs, un document contient à la fois les données et leur structure. Il peut contenir des valeurs simples, des tableaux, et des documents imbriqués.

JSON — Document produit SportShop { "_id": "prod_001", "nom": "Chaussures Trail X200", "prix": 129.99, "categorie": "running", "specs": { // ← document imbriqué "tailles": [39, 40, 41, 42], "couleur": "noir/rouge", "drop": "6mm" }, "avis": [ // ← tableau de sous-documents { "note": 5, "auteur": "Alice" }, { "note": 4, "auteur": "Bob" } ] }
🏗️
La force du schéma flexible

Dans une même collection (l'équivalent d'une table), des documents peuvent avoir des structures différentes. C'est ce qu'on appelle le schéma flexible ou schemaless. Pour un catalogue e-commerce, c'est une révolution : un t-shirt et un smartphone peuvent cohabiter dans la même collection sans gaspiller d'espace pour des champs vides.

JSON — Deux documents dans la même collection // Document 1 — T-shirt { "nom": "T-shirt SportShop", "taille": "M", "matiere": "polyester" } // Document 2 — Montre GPS (champs complètement différents) { "nom": "Montre GPS Pro", "autonomie": "20h", "gps": true, "etanche": "50m" }
En SQL, vous auriez eu une table avec des dizaines de colonnes nullable — la plupart vides. En NoSQL documentaire, chaque document ne contient que ce dont il a besoin.
⚖️
Embedding vs Referencing — la décision clé

C'est la question centrale de la modélisation documentaire : doit-on intégrer les données liées dans le document (embedding) ou stocker une référence vers un autre document (referencing) ?

📦 EMBEDDING — Imbrication
Les données liées sont stockées directement à l'intérieur du document principal. Un seul appel en base suffit pour tout récupérer.
✓ Lecture rapide ✓ Requête simple × Documents volumineux × Mise à jour complexe
🔗 REFERENCING — Référence
Les documents stockent des identifiants qui pointent vers d'autres documents. Similaire aux clés étrangères SQL.
✓ Pas de duplication ✓ Mise à jour centralisée × Plusieurs requêtes × Jointures manuelles
Règle pragmatique : si les données sont lues ensemble et rarement mises à jour indépendamment → embedding. Si les données sont partagées entre plusieurs documents et mises à jour fréquemment → referencing.
🏭

📰

  • Documents trop imbriqués = complexité
1.5

MongoDB — L'outil de référence

MongoDB est aujourd'hui la base documentaire la plus utilisée dans l'industrie. Des startups aux multinationales (eBay, Cisco, Electronic Arts…), elle s'est imposée grâce à son équilibre entre flexibilité, performance et richesse fonctionnelle. Comprendre son architecture, c'est comprendre le modèle document en action.

🏗️
Architecture de MongoDB — La hiérarchie

MongoDB s'organise selon une hiérarchie simple que vous pouvez rapprocher du monde SQL pour la comprendre rapidement.

🗄️
Base de données
≈ Database SQL
📂
Collection
≈ Table SQL (sans schéma)
📄
Document
≈ Ligne SQL (flexible)
🔡
Champ
≈ Colonne SQL (typée)
📝
Le format BSON — JSON binaire étendu

MongoDB ne stocke pas du JSON pur, mais du BSON (Binary JSON). La différence ? BSON ajoute des types que JSON ne supporte pas nativement et est optimisé pour la lecture/écriture rapide.

BSON — Types étendus MongoDB { "_id": ObjectId("64f3a2b1c9e..."), // ← ID unique auto-généré "nom": "Trail X200", "prix": 129.99, // ← Double "stock": 42, // ← Int32 "disponible": true, // ← Boolean "createdAt": ISODate("2024-09-01") // ← Date native }
🔍
Requêtes MongoDB — La philosophie

Les requêtes MongoDB sont elles-mêmes des documents JSON. Pas de syntaxe SQL à apprendre — vous décrivez ce que vous cherchez avec la même structure que vos données.

MongoDB — Exemples de requêtes (lecture seule ici) // Trouver tous les produits de la catégorie "running" db.produits.find({ "categorie": "running" }) // Produits entre 50€ et 150€, triés par prix db.produits.find({ "prix": { $gte: 50, $lte: 150 } }) .sort({ "prix": 1 }) // Produits en stock ET note moyenne > 4 db.produits.find({ $and: [ { "stock": { $gt: 0 } }, { "note_moyenne": { $gt: 4 } } ]})
$gte, $lte, $gt, $lt — opérateurs de comparaison. $and, $or, $not — opérateurs logiques. Vous les retrouverez dans tous les drivers MongoDB (Node.js, Python, Java…).
📊
L'indexation — pourquoi c'est crucial

Sans index, MongoDB scanne tous les documents de la collection pour trouver les résultats. Avec des millions de documents, c'est catastrophique. Un index crée une structure optimisée qui accélère drastiquement les requêtes.

  • Index simple — sur un seul champ : db.produits.createIndex({"prix": 1})
  • Index composé — sur plusieurs champs combinés, pour les requêtes complexes
  • Index textuel — pour la recherche full-text dans les descriptions
  • Index TTL — expire automatiquement les documents après N secondes (sessions, tokens…)
  • Index géospatial — pour les requêtes de proximité géographique
🌩️
L'écosystème MongoDB
MongoDB Atlas
Service cloud géré (AWS/GCP/Azure). Déploiement en quelques clics, scaling automatique, backups inclus. C'est ce que vous utiliserez en TP.
MongoDB Compass
Interface graphique officielle. Visualisation des données, construction de requêtes, analyse des performances et des index.
Mongoose (Node.js)
ODM (Object-Document Mapper) pour Node.js. Ajoute de la structure via des Schemas, simplifie la connexion et les opérations CRUD.
MongoDB Shell (mongosh)
CLI interactive pour exécuter des requêtes directement. JavaScript natif. Incontournable pour le débogage et l'administration.

  • 4 modèles existent : Clé-Valeur, Document, Colonne, Graphe — chacun a son domaine d'excellence
