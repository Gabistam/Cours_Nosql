🕰️ Naissance du modèle Document

1970–2000
Le règne du relationnel. Oracle, MySQL, PostgreSQL dominent. Tout est normalisé en tables, les relations se font par clés étrangères et JOIN. Efficace pour la cohérence, mais rigide face aux données semi-structurées.
2000–2007
L'explosion du Web 2.0. Amazon, Google, Facebook gèrent des milliards d'objets hétérogènes (profils, posts, produits). Les schémas fixes deviennent un frein : chaque produit Amazon a des attributs différents (livres ≠ vêtements ≠ électronique).
2007
10gen crée MongoDB (renommé MongoDB Inc. en 2013). L'idée : stocker les données comme elles existent naturellement dans le code — des objets JSON. Plus de mapping objet-relationnel complexe.
2009
MongoDB 1.0 sort en open source. Le concept de "schéma flexible" séduit les startups qui ont besoin d'itérer vite sans migrations de tables.
2016–auj.
MongoDB Atlas lance le cloud managé (M0 gratuit). Aujourd'hui, MongoDB est la base NoSQL document la plus utilisée au monde, avec 40 000+ entreprises clientes.

🧩 Document vs Relationnel

Un produit dans un schéma relationnel classique est éclaté sur plusieurs tables. Dans MongoDB, tout tient dans un seul document.

🗃️ Relationnel — 4 tables

SQL · structure
-- 4 tables pour un produit :
products (id, name, price, category_id)
categories (id, name)
product_specs (product_id, key, value)
reviews (id, product_id, user, rating, comment)

-- Pour lire un produit complet :
SELECT p.*, c.name, s.key, s.value,
       r.user, r.rating, r.comment
FROM products p
JOIN categories c ON p.category_id = c.id
JOIN product_specs s ON s.product_id = p.id
JOIN reviews r ON r.product_id = p.id
WHERE p.id = 'p-run-001'

🍃 MongoDB — 1 document

{
  "id": "p-run-001",
  "name": "Nike Air Zoom Pegasus 40",
  "category": "running",
  "price": 129.99,
  "tags": ["route", "amorti"],
  "specs": {
    "drop": "10mm",
    "poids": "283g"
  },
  "reviews": [
    { "author": "Marie L.", "rating": 5 },
    { "author": "Thomas B.", "rating": 4 }
  ],
  "avgRating": 4.7
}
Dénormalisation intentionnelle : dans MongoDB on accepte la duplication de données (ex: le texte d'une review est dans le produit) pour éviter les JOIN. C'est un choix de performance — une seule lecture disque suffit pour tout le document.

🔑 JSON vs BSON

📝
JSON
Format texte lisible par l'humain
  • String, Number, Boolean, Array, Object, null
  • Format d'échange universel (API, fichiers)
  • Ce que tu écris dans data/products.json
  • Pas de type Date natif — tout est string
🗄️
BSON
Binary JSON — format de stockage MongoDB
  • Superset binaire de JSON
  • Types supplémentaires : Date, ObjectId, Decimal128, Binary
  • Stockage compact, lecture rapide
  • Mongoose convertit automatiquement JSON ↔ BSON
En pratique : tu n'interagis jamais directement avec BSON. Tu manipules du JSON dans ton code JavaScript, Mongoose/MongoDB font la conversion en arrière-plan. Retiens juste que _id est un ObjectId BSON et que new Date() crée un vrai type Date, pas une string.

🔓 Le schéma flexible — avantage clé

Dans SportShopy, chaque catégorie de produit a des specs différentes. En SQL, il faudrait une table specs avec des colonnes génériques ou pire, une colonne par attribut possible. En MongoDB, le champ specs est simplement un objet Mixed :

document · running
"specs": {
  "pointure": "38-48",
  "drop": "10mm",
  "typeFoulee": "universelle",
  "poids": "283g"
}
document · natation
"specs": {
  "taille": "XS-XL",
  "materiau": "polyester",
  "protection": "UPF 50+"
}
Même collection, specs totalement différentes — MongoDB stocke les deux sans problème. Le schéma Mongoose utilise mongoose.Schema.Types.Mixed pour ce champ, ce qui permet n'importe quelle structure.

// ── TROUVER ────────────────────────────────────────────
db.products.find({ category: "running" })

// ── COMPARAISON ────────────────────────────────────────

🦴 Mongoose — pourquoi l'utiliser ?

🦴
ODM — Object Document Mapper

MongoDB est schéma-free par nature. Mongoose ajoute une couche de validation côté application : types, champs requis, valeurs acceptées, index automatiques. Sans Mongoose tu pourrais insérer n'importe quoi.

Sans Mongoose (driver brut)

javascript · risqué
// Aucune validation — n'importe quoi passe
await db.collection('products').insertOne({
  name: 42, // number au lieu de string
  price: "cher", // string au lieu de number
  category: "truc" // valeur invalide
})
// → inséré sans erreur 😱

Avec Mongoose (ODM)

javascript · sécurisé
// Mongoose valide avant d'envoyer
await new Product({
  name: 42,
  price: "cher",
  category: "truc"
}).save()
// → ValidationError: category must be
// one of [running, football, ...]

Étape 3 — Modèle Mongoose Product

🧬
server/models/Product.js
Schéma avec sous-documents, index et validation
Mongoose apporte la validation que MongoDB n'impose pas nativement. Le champ specs est Mixed pour accepter les attributs différents par catégorie (drop pour running, matériau pour natation, etc.).
javascript · server/models/Product.js
const mongoose = require('mongoose')

// Sous-schéma pour les avis (pas de _id propre)
const reviewSchema = new mongoose.Schema({
  author: { type: String, required: true },
  rating: { type: Number, required: true, min: 1, max: 5 },
  comment: { type: String, required: true },
  date: { type: String, required: true },
}, { _id: false })

const productSchema = new mongoose.Schema({
  id: { type: String, required: true, unique: true },
  name: { type: String, required: true },
  category: { type: String, required: true,
                   enum: ['running','football','fitness','natation','cyclisme','tennis'] },
  price: { type: Number, required: true },
  originalPrice: { type: Number },
  stock: { type: Number, required: true, default: 0 },
  image: { type: String, required: true },
  tags: [String],
  specs: { type: mongoose.Schema.Types.Mixed, default: {} },
  reviews: [reviewSchema],
  avgRating: { type: Number, default: 0 },
}, {
  timestamps: { createdAt: 'createdAt', updatedAt: false },
})

// Index full-text pour la recherche
productSchema.index({ name: 'text', tags: 'text' })
// Index de tri fréquent
productSchema.index({ category: 1, price: 1 })
productSchema.index({ avgRating: -1 })

module.exports = mongoose.model('Product', productSchema)

server/middleware/validateQuery.js — valider les paramètres

javascript · server/middleware/validateQuery.js
const VALID_CATEGORIES = ['running','football','fitness','natation','cyclisme','tennis']
const VALID_SORT = ['price_asc','price_desc','rating','newest']

function validateProductQuery(req, res, next) {
  const { category, minPrice, maxPrice, minRating, sort } = req.query

  if (category && !VALID_CATEGORIES.includes(category))
    return res.status(400).json({ success: false, error: `Catégorie invalide` })
  if (sort && !VALID_SORT.includes(sort))
    return res.status(400).json({ success: false, error: `Tri invalide` })

  next()
}

module.exports = { validateProductQuery }

server/services/productService.js

La logique métier complète : construction dynamique du filtre MongoDB, map de tri, gestion de l'erreur 404 avec un statut HTTP explicite.

javascript · solution complète
const Product = require('../models/Product')

async function getAll(filters = {}) {
  const { category, search, minPrice, maxPrice, minRating, sort } = filters
  const query = {}

  if (category) query.category = category
  if (minRating) query.avgRating = { $gte: Number(minRating) }
  if (minPrice || maxPrice) {
    query.price = {}
    if (minPrice) query.price.$gte = Number(minPrice)
    if (maxPrice) query.price.$lte = Number(maxPrice)
  }
  if (search) query.$text = { $search: search }

  const sortMap = {
    price_asc: { price: 1 },
    price_desc: { price: -1 },
    rating: { avgRating: -1 },
    newest: { createdAt: -1 },
  }
  const sortOption = sortMap[sort] || { createdAt: -1 }

  const products = await Product.find(query).sort(sortOption).lean()
  return { products, total: products.length }
}

async function getById(id) {
  const product = await Product.findOne({ id }).lean()
  if (!product) {
    const err = new Error(`Produit introuvable : ${id}`)
    err.status = 404
    throw err
  }
  return product
}

async function getFeatured() {
  return Product.find().sort({ avgRating: -1 }).limit(4).lean()
}

module.exports = { getAll, getById, getFeatured }
.lean() : retourne des objets JavaScript purs au lieu d'instances Mongoose. Plus léger et ~2× plus rapide en lecture — à utiliser dès qu'on n'a pas besoin des méthodes du modèle (save, validate…).

Glossaire MongoDB

Les 20 termes incontournables pour parler MongoDB sans hésiter. Chaque terme que tu rencontres dans le code ou la doc est défini ici.

01Document
Unité de base du stockage MongoDB. Un objet JSON/BSON auto-descriptif qui peut contenir des valeurs scalaires, des tableaux et des objets imbriqués. Équivalent d'une ligne en SQL, mais sans contrainte de structure uniforme.
modèle
02Collection
Groupe de documents dans une même base de données. Équivalent d'une table SQL, mais sans schéma imposé — chaque document de la collection peut avoir des champs différents.
modèle
03BSON
Binary JSON — le format de stockage interne de MongoDB. Superset binaire de JSON qui ajoute des types natifs : Date, ObjectId, Decimal128, Binary. Plus compact et plus rapide à lire que du texte pur.
concept
04ObjectId
Identifiant unique sur 12 octets généré automatiquement par MongoDB pour le champ _id. Encode horodatage + machine + processus + compteur — garantit l'unicité sans coordination entre serveurs.
concept
05Schéma (Mongoose)
Définition de la structure attendue d'un document, imposée côté application par Mongoose. Permet de typer les champs, les rendre obligatoires, les valider (enum, min/max) et créer des index automatiquement.
outil
06Index
Structure de données qui accélère les requêtes sur un ou plusieurs champs. Sans index, MongoDB parcourt tous les documents (collection scan). Avec index, la recherche est logarithmique. À créer sur les champs fréquemment filtrés ou triés.
requête
07Index full-text
Index spécial déclaré avec 'text' qui tokenise les chaînes pour permettre la recherche par mots-clés via $text: { $search: "..." }. Dans SportShopy, il couvre les champs name et tags.
requête
08Aggregation Pipeline
Série d'étapes de transformation ($match, $group, $sort, $project…) appliquées séquentiellement à une collection. Équivalent des requêtes SQL avec GROUP BY et fonctions d'agrégation.
requête
09Embedding
Stratégie de modélisation : stocker les données liées à l'intérieur du même document. Ex : les reviews dans le document produit. Évite les JOIN, optimise la lecture. Recommandé quand les données liées sont toujours lues ensemble.
modèle
10Referencing
Stratégie alternative : stocker un ObjectId qui pointe vers un autre document (dans la même collection ou une autre). Nécessite une deuxième requête pour récupérer les données liées (populate() en Mongoose).
modèle
11ODM
Object Document Mapper. Couche logicielle qui fait le pont entre les objets JavaScript et les documents MongoDB. Mongoose est l'ODM le plus utilisé pour Node.js — il ajoute validation, hooks et requêtes chainables.
outil
12Atlas
Service cloud managé de MongoDB (DBaaS). Gère les backups, la réplication, le monitoring et les mises à jour. Le tier M0 gratuit (512 Mo) est suffisant pour les projets pédagogiques et les prototypes.
infra
13Replica Set
Groupe de 3 serveurs MongoDB synchronisés : 1 primaire (lectures/écritures) + 2 secondaires (copies). Si le primaire tombe, une élection automatique désigne un nouveau primaire. Garantit la haute disponibilité.
infra
14Sharding
Partitionnement horizontal des données sur plusieurs serveurs (shards) selon une clé de partition. Permet de gérer des volumes de données qui dépassent la capacité d'un seul serveur. Activé manuellement sur les très grands datasets.
infra
15.lean()
Option Mongoose qui retourne des objets JavaScript purs au lieu d'instances de modèle enrichies. Environ 2× plus rapide en lecture car Mongoose n'attache pas les méthodes, getters et setters. À utiliser dès qu'on n'a pas besoin de .save().
outil
16populate()
Méthode Mongoose qui remplace un ObjectId par le document référencé en effectuant une deuxième requête automatique. Équivalent d'un JOIN limité — pratique pour le referencing, mais à éviter dans les boucles (N+1 queries).
outil
17Middleware Mongoose
Fonctions pre / post attachées à certaines opérations du modèle (save, find, deleteOne…). Utiles pour hacher un mot de passe avant sauvegarde, déclencher un log, ou calculer automatiquement un champ dérivé.
outil
18Virtual
Champ Mongoose calculé à la volée, non stocké en base. Ex : fullName = firstName + lastName. Accessible comme une propriété normale de l'instance mais invisible dans Atlas. Permet d'enrichir le modèle sans dupliquer de données.
outil
19TTL Index
Index spécial qui supprime automatiquement les documents après un délai défini (en secondes). Déclaré avec { expireAfterSeconds: N } sur un champ de type Date. Idéal pour les sessions, tokens temporaires, logs à durée limitée.
requête
20$lookup
Étape d'aggregation pipeline qui effectue une jointure entre deux collections (équivalent du LEFT JOIN SQL). Retourne les documents d'une collection enrichis avec les données correspondantes d'une autre. Plus lourd qu'un embedding, à réserver aux rapports complexes.
requête
