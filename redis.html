Le modèle Clé-Valeur

Redis stocke tout en RAM sous forme de paires clé→valeur. Pas de schéma, pas de jointure, pas de disque sur le chemin critique. Résultat : des latences en microsecondes là où MongoDB prend des millisecondes.

⚡ Pourquoi Redis pour le panier ?

Le panier d'achat a des exigences très différentes du catalogue produits :

CritèreMongoDB (catalogue)Redis (panier)
Durée de viePermanent (catalogue stable)Temporaire — 24h max
Latence~2–10 ms< 1 ms
StructureDocuments complexes imbriquésPaire simple clé → JSON
RelationsCatalogue, catégories, avisAucune — sessionId→panier
ExpirationManuelle (champ date)Natif (TTL)
Volume20 produits (stable)1 clé par session active
Règle empirique : Redis quand les données sont éphémères (sessions, paniers, cache, rate limiting) et que la latence est critique. MongoDB quand les données sont persistantes et structurées.

🗂️ Les 5 types de données Redis

📝
String
Le type de base — utilisé dans SportShopy

Une clé → une valeur texte (ou JSON sérialisé). C'est le type qu'on utilise pour le panier : on sérialise le tableau d'items en JSON et on le stocke dans une clé cart:{sessionId}.

redis cli
SET cart:abc123 '{"items":[]}' EX 86400
GET cart:abc123
TTL cart:abc123 # secondes restantes
🗃️
Hash
Objet structuré (champs:valeurs)

Idéal pour les profils utilisateurs ou sessions enrichies. Chaque champ est accessible individuellement sans désérialiser tout l'objet.

redis cli
HSET user:42 name "Alice" email "a@b.com"
HGET user:42 name
HGETALL user:42
📋
List
File / pile ordonnée

Liste doublement chaînée. Parfaite pour les files d'attente (FIFO avec LPUSH + RPOP) ou les historiques d'activité.

redis cli
LPUSH queue:emails "msg1" "msg2"
RPOP queue:emails # dépile FIFO
LRANGE queue:emails 0 -1
🔢
Sorted Set
Ensemble ordonné par score

Ensemble de membres triés par un score numérique. Idéal pour les classements, leaderboards, ou tags populaires.

redis cli
ZADD leaderboard 1500 "Alice"
ZADD leaderboard 2300 "Bob"
ZRANGE leaderboard 0 -1 WITHSCORES REV

💾 Persistance — RDB vs AOF

Redis est in-memory mais propose deux mécanismes de persistance pour survivre aux redémarrages :

MécanismeCommentAvantageInconvénient
RDB (snapshot)Dump binaire périodiqueCompact, rapide au redémarragePerd les données depuis le dernier snapshot
AOF (Append Only File)Log de toutes les écrituresPerte de données minimaleFichier plus volumineux
Les deuxRDB + AOF activésMeilleure garantieConfig plus lourde
Dans SportShopy, le docker-compose utilise --appendonly yes (AOF activé). Pour un panier, perdre quelques secondes de données est acceptable — mais c'est une bonne habitude de l'activer dès le dev.

Solutions complètes

Code final validé. Essaie d'abord — ne regarde ici qu'en cas de blocage.

server/services/cartService.js

javascript · solution complète
const redis = require('../config/redis')

const TTL = 60 * 60 * 24 // 24h
const key = (sid) => `cart:${sid}`

async function getCart(sessionId) {
  const raw = await redis.get(key(sessionId))
  return _withTotals(raw ? JSON.parse(raw) : { items: [] })
}

async function addToCart(sessionId, product, qty = 1) {
  const cart = await _load(sessionId)
  const existing = cart.items.find(i => i.productId === product.productId)
  if (existing) existing.quantity += qty
  else cart.items.push({ ...product, quantity: qty })
  await _save(sessionId, cart)
  return _withTotals(cart)
}

async function updateQuantity(sessionId, productId, quantity) {
  const cart = await _load(sessionId)
  if (quantity <= 0) cart.items = cart.items.filter(i => i.productId !== productId)
  else { const item = cart.items.find(i => i.productId === productId); if (item) item.quantity = quantity }
  await _save(sessionId, cart)
  return _withTotals(cart)
}

async function removeFromCart(sessionId, productId) {
  const cart = await _load(sessionId)
  cart.items = cart.items.filter(i => i.productId !== productId)
  await _save(sessionId, cart)
  return _withTotals(cart)
}

async function clearCart(sessionId) {
  await redis.del(key(sessionId))
  return { items: [], total: 0, count: 0 }
}

async function getCartSummary(sessionId) {
  const raw = await redis.get(key(sessionId))
  if (!raw) return { count: 0, total: 0 }
  const cart = JSON.parse(raw)
  return {
    count: cart.items.reduce((s,i) => s + i.quantity, 0),
    total: +cart.items.reduce((s,i) => s + i.price * i.quantity, 0).toFixed(2),
  }
}

async function _load(sid) {
  const raw = await redis.get(key(sid))
  return raw ? JSON.parse(raw) : { items: [] }
}
async function _save(sid, cart) {
  await redis.set(key(sid), JSON.stringify(cart), 'EX', TTL)
}
function _withTotals(cart) {
  return {
    items: cart.items,
    count: cart.items.reduce((s,i) => s + i.quantity, 0),
    total: +cart.items.reduce((s,i) => s + i.price * i.quantity, 0).toFixed(2),
  }
}

module.exports = { getCart, addToCart, updateQuantity, removeFromCart, clearCart, getCartSummary }
Pourquoi sérialiser en JSON String et non Hash Redis ? Le panier est toujours lu/écrit en entier. Une String JSON est plus simple et évite les commandes HSET/HGET multiples. Un Hash serait pertinent si on lisait souvent un seul champ.

⚡ Cheatsheet Redis + ioredis

Redis CLI · Cheatsheet
// ── LECTURE/ÉCRITURE ──────────────────────────────────
SET key value EX seconds // stocker avec TTL
GET key // lire
DEL key // supprimer
EXISTS key // 1 si existe, 0 sinon

// ── TTL ──────────────────────────────────────────────
TTL key // secondes restantes (-1=no TTL, -2=inexistant)
EXPIRE key seconds // renouveler le TTL
PERSIST key // supprimer le TTL

// ── EXPLORATION ──────────────────────────────────────
KEYS pattern // ex: KEYS cart:* (pas en prod)
SCAN 0 MATCH cart:* COUNT 100 // version safe pour la prod
DBSIZE // nombre de clés total

// ── IOREDIS ──────────────────────────────────────────
await redis.set(key, value, 'EX', ttl)
await redis.get(key) // null si inexistant
await redis.del(key)
await redis.ttl(key)

Glossaire Redis

Les 20 termes incontournables pour parler Redis sans hésiter.

01In-memory
Redis stocke toutes les données en RAM. Pas de lecture disque sur le chemin critique → latences en microsecondes. Contrepartie : la capacité est limitée par la mémoire disponible et les données sont volatiles sans persistance activée.
concept
02TTL
Time To Live — durée en secondes avant expiration automatique d'une clé. Redis supprime la clé sans intervention externe. Dans SportShopy, le panier expire après 86 400 s (24h) d'inactivité. Commandes : EXPIRE key n / option EX n du SET.
commande
03Clé-Valeur
Modèle de stockage le plus simple : une chaîne de caractères unique (clé) pointe vers une valeur. La clé est arbitraire — par convention on utilise des séparateurs (:) pour créer des espaces de noms : cart:abc, user:42:prefs.
concept
04String
Type Redis de base. Valeur jusqu'à 512 Mo. Peut contenir du texte, du JSON sérialisé, des nombres entiers (atomiquement incrémentés avec INCR), ou des données binaires. C'est le type utilisé pour le panier dans SportShopy.
type
05Hash
Objet Redis avec plusieurs champs : HSET key field value. Idéal pour les profils utilisateurs ou sessions enrichies. Permet de lire/écrire un seul champ sans désérialiser tout l'objet — avantage sur la String JSON quand les champs sont souvent accédés séparément.
type
06List
Liste doublement chaînée. LPUSH / RPUSH pour ajouter, LPOP / RPOP pour retirer. Parfaite pour les files d'attente FIFO (LPUSH + RPOP), les historiques ou les flux d'activité récente.
type
07Set
Ensemble de chaînes sans doublons. SADD / SMEMBERS / SISMEMBER. Utile pour les tags uniques, les utilisateurs connectés, ou les systèmes de vote (un utilisateur = un vote).
type
08Sorted Set
Set ordonné par un score numérique. ZADD / ZRANGE / ZRANK. Parfait pour les classements, leaderboards, ou tags populaires triés par fréquence d'utilisation.
type
09EXPIRE / EX
EXPIRE key seconds définit ou renouvelle le TTL d'une clé existante. L'option EX de SET combine création et TTL en une commande atomique : SET key val EX 3600. Dans cartService, chaque écriture renouvelle le TTL (comportement "sliding window").
commande
10RDB
Redis Database Backup — snapshot binaire compressé de toutes les données, sauvegardé périodiquement sur disque. Rapide au redémarrage mais peut perdre les données depuis le dernier snapshot. Activé par défaut dans Redis.
concept
11AOF
Append Only File — log de toutes les commandes d'écriture. Reconstruit l'état exact lors d'un redémarrage. Perte de données minimale (dernière seconde). Activé dans SportShopy via --appendonly yes dans docker-compose.
infra
12Session
Identifiant unique associant un visiteur à ses données côté serveur. Dans SportShopy, le sessionId est un UUID généré côté client (crypto.randomUUID()) et stocké dans localStorage. Il est transmis via le header x-session-id.
concept
13Cache
Stocker temporairement le résultat d'une opération coûteuse pour éviter de la recalculer. Redis est la solution de cache la plus utilisée : résultats de requêtes MongoDB, réponses d'API externes, pages HTML générées. TTL = durée de validité du cache.
concept
14ioredis
Client Redis le plus populaire pour Node.js. Supporte les promesses nativement, la reconnexion automatique, les pipelines, les clusters et Sentinel. Alternative : redis (client officiel), plus simple mais moins de fonctionnalités avancées.
outil
15Redis Commander
Interface web open-source pour explorer et modifier les données Redis. Visualise les clés, valeurs, TTL et types. Lancé via Docker dans SportShopy sur le port 8081. À ne jamais exposer en production.
outil
16Pub/Sub
Mécanisme de messagerie Redis : un publisher envoie sur un canal (PUBLISH), des subscribers reçoivent en temps réel (SUBSCRIBE). Utile pour les notifications push, le chat, la synchronisation entre microservices. Non utilisé dans cette phase mais courant en production.
concept
17Pipeline
Mécanisme ioredis qui regroupe plusieurs commandes en un seul aller-retour réseau. Au lieu d'envoyer 5 commandes séparément (5 allers-retours), on les pipeline et on attend une seule réponse groupée. Gain de latence significatif pour les opérations en lot.
outil
18Atomicité
Redis exécute chaque commande de manière atomique — elle ne peut pas être interrompue à mi-chemin par une autre commande. INCR par exemple lit ET incrémente en une seule opération. Pour grouper plusieurs commandes atomiques : MULTI / EXEC (transactions Redis).
concept
19Eviction Policy
Comportement de Redis quand la mémoire est pleine. Politiques courantes : allkeys-lru (supprime les clés les moins récemment utilisées), volatile-lru (idem mais seulement les clés avec TTL), noeviction (erreur en écriture). Configuré via maxmemory-policy.
infra
20SCAN
Alternative à KEYS * pour la production. SCAN cursor MATCH pattern COUNT n itère sur les clés par lots sans bloquer Redis. KEYS * scanne toutes les clés d'un coup et peut bloquer le serveur sur un gros dataset — à éviter en prod.
commande
